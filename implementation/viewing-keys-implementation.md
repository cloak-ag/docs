## Overview

This document describes the v2.0 implementation of viewing keys for compliance in Cloak. The key difference from v1.0:

- **v1.0**: Master compliance key at relay, derived user keys
- **v2.0**: User generates viewing key locally, sends to relay for storage

This makes Cloak's model closer to ZCash, with the only difference being that viewing keys are **mandatory** (auto-shared with relay).

---

## The Problem & Solution

### The Problem

In v1.0:
- Master compliance key was generated by relay
- User-specific keys were derived from master + wallet address
- Relay held the master private key (could decrypt everything)

### The Solution (v2.0)

```
┌─────────────────────────────────────────────────────────────────────────┐
│ BEFORE (v1.0)                                                           │
│                                                                          │
│  Master Key (relay) ──derive──► User Key ──encrypt──► Encrypted Data   │
│       │                                                                     │
│       └────── Can decrypt ALL user data!                                  │
└─────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────────┐
│ AFTER (v2.0)                                                            │
│                                                                          │
│  User generates locally ──send to relay──► Stored in DB                 │
│       │                                                                     │
│       └────── Relay can decrypt (has user's viewing key)                  │
│                                                                          │
│  Same as ZCash when user shares viewing key with auditor                │
└─────────────────────────────────────────────────────────────────────────┘
```

## Key Generation

### SDK: Generate Viewing Keypair

**File**: `sdk/src/core/compliance-keys.ts`

```typescript
import nacl from "tweetnacl";

export interface ViewingKeyPair {
  privateKey: Uint8Array;  // 32 bytes - for decryption
  publicKey: Uint8Array;   // 32 bytes - for encryption
}

export function generateViewingKeyPair(): ViewingKeyPair {
  const keyPair = nacl.box.keyPair();
  return {
    privateKey: keyPair.secretKey,
    publicKey: keyPair.publicKey,
  };
}
```

### Web: Get Viewing Key from Wallet

**File**: `web/lib/metadata-history.ts`

The web app uses the existing Cloak wallet keys (`keys.view.vk_secret`) as the viewing key:

```typescript
const keys = await getOrCreateWalletKeysAsync();
const viewingKey = {
  privateKey: keys.view.vk_secret,  // Already have this from wallet
  publicKey: keys.view.pvk,
};
```

## Transaction Flow

### First Transaction (Registration)

When user makes their first deposit/withdraw/swap:

```
1. SDK creates metadata bundle WITH viewing_key:
   {
     encrypted_user: "base64...",
     encrypted_compliance: "base64...",
     user_pubkey: "...",
     commitment: "...",
     timestamp: 1234567890,
     tx_type: "deposit",
     viewing_key: "a1b2c3..."  // ONLY on first tx!
   }

2. POST to /transact or /withdraw

3. Relay receives:
   - Validates wallet signature
   - Checks if viewing_key exists for user
   - If NOT exists: saves viewing_key to database
   - Stores encrypted metadata blob

4. Mark viewing_key as registered (in memory)
```

### Subsequent Transactions

```
1. SDK creates metadata bundle WITHOUT viewing_key:
   {
     encrypted_user: "base64...",
     encrypted_compliance: "base64...",
     user_pubkey: "...",
     commitment: "...",
     timestamp: 1234567890,
     tx_type: "withdraw"
     // NO viewing_key
   }

2. POST to /transact or /withdraw

3. Relay:
   - Validates wallet signature
   - Looks up viewing_key from database
   - Stores encrypted blob
```

## Compliance Decryption

### Web Client: Encrypt for Compliance

**File**: `web/lib/metadata-history.ts`

```typescript
import nacl from "tweetnacl";

function encryptForRecipient(
  plaintext: Uint8Array,
  recipientPublicKey: Uint8Array
): EncryptedMetadataPayload {
  const ephemeral = nacl.box.keyPair();
  const sharedSecret = nacl.scalarMult(ephemeral.secretKey, recipientPublicKey);
  const nonce = randomBytes(12);
  const ciphertext = aesGcmEncrypt(plaintext, sharedSecret, nonce);

  return {
    ephemeral_pk: hex(ephemeral.publicKey),
    ciphertext: hex(ciphertext),
    nonce: hex(nonce),
  };
}
```

### Relay: Decrypt for Compliance

**File**: `services/relay/src/api/compliance.rs`

```rust
fn decrypt_with_viewing_key(
    encrypted_base64: &str,
    viewing_private_key: [u8; 32],
) -> Result<DecryptedTransactionMetadata, Error> {
    // 1. Decode base64 to get payload
    let payload: EncryptedMetadataPayload = decode(encrypted_base64)?;
    
    // 2. Compute shared secret: X25519(ephemeral_pk, viewing_private_key)
    let shared_secret = MontgomeryPoint(ephemeral_pk)
        .mul_clamped(viewing_private_key)
        .to_bytes();
    
    // 3. Decrypt with AES-256-GCM
    let plaintext = aes_decrypt(ciphertext, shared_secret, nonce)?;
    
    // 4. Parse JSON
    Ok(serde_json::from_slice(plaintext)?)
}
```

### Admin Query Flow

```
1. Admin signs message: "CLOAK_ADMIN_{target_user}"
2. POST /admin/compliance/decrypt { user_pubkey, admin_wallet, signature }
3. Relay:
   - Validates admin wallet
   - Looks up viewing_key for user from DB
   - For each encrypted blob: decrypt_with_viewing_key()
   - Returns plaintext transactions
4. Admin sees decrypted data
```

## Database Schema

### Table: viewing_keys

```sql
CREATE TABLE viewing_keys (
    user_pubkey VARCHAR(44) PRIMARY KEY,
    viewing_key VARCHAR(64) NOT NULL,  -- hex-encoded 32 bytes
    created_at TIMESTAMP DEFAULT NOW()
);
```

### Table: transaction_metadata

```sql
CREATE TABLE transaction_metadata (
    id SERIAL PRIMARY KEY,
    user_pubkey VARCHAR(44) NOT NULL,
    commitment VARCHAR(128) NOT NULL,
    encrypted_user TEXT NOT NULL,
    encrypted_compliance TEXT NOT NULL,
    tx_type VARCHAR(20) NOT NULL,
    timestamp_ms BIGINT NOT NULL,
    created_at TIMESTAMP DEFAULT NOW()
);

CREATE INDEX idx_metadata_user_pubkey ON transaction_metadata(user_pubkey);
CREATE INDEX idx_metadata_timestamp ON transaction_metadata(timestamp_ms);
```

## Summary

### What Was Implemented

1. **Local Key Generation**: User generates X25519 keypair in browser
2. **Auto-Registration**: Viewing key sent on first transaction, stored at relay
3. **Encrypted Storage**: Transaction metadata encrypted with user's viewing key
4. **Relay Decryption**: Relay uses stored viewing key for compliance queries
5. **Admin API**: Authorized admins can decrypt any user's transactions

### Key Differences from v| Aspect | v1.0

1.0 | v2.0 |
|--------|------|-------|
| Key Origin | Master key at relay | User generates locally |
| Key Storage | Master key (single point) | Per-user viewing key |
| Encryption | Double (user + master) | Single (user only) |
| Relay Power | Can decrypt all | Can decrypt (has key) |

### Similar to ZCash

v2.0 is now closer to ZCash's model:
- User generates viewing key locally
- User can share viewing key with third party
- Third party can decrypt transactions

**Difference**: In Cloak, sharing is mandatory (relay always has the key).

### Security Trade-off

- **Before**: Single master key = single point of failure
- **After**: Each user has their own key = distributed trust

Users must trust the relay because it stores their viewing key, but this is the same as ZCash when sharing with an auditor.
